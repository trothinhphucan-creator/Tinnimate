## 🔴 Critical Issues (Phải sửa ngay)

### 1. MoMo Callback KHÔNG xác minh chữ ký (Giả mạo thanh toán)

- **File:** [momo/callback/route.ts](file:///home/haichu/tinnimate/web/app/api/payment/momo/callback/route.ts)
- **Nguy hiểm:** Bất kỳ ai cũng có thể gửi request giả đến `/api/payment/momo/callback` với `resultCode=0` và `extraData` tùy ý → **tự nâng cấp tài khoản lên Premium/Pro/Ultra miễn phí**
- **So sánh:** VNPay callback đã verify HMAC-SHA512 đúng cách. Stripe Webhook đã verify signature đúng cách. Chỉ MoMo bị bỏ sót.
- **Cách sửa:** Thêm verify HMAC-SHA256 signature giống như lúc tạo request. So sánh signature từ MoMo gửi về với signature tính từ response params + `secretKey`.

```typescript
// Cần thêm verify signature trước khi xử lý
const rawSig = `accessKey=${accessKey}&amount=${body.amount}&extraData=${body.extraData}&...`
const expectedSig = crypto.createHmac('sha256', secretKey).update(rawSig).digest('hex')
if (body.signature !== expectedSig) {
  return Response.json({ error: 'Invalid signature' }, { status: 400 })
}
```

### 2. Guest Chat Rate Limit dựa vào Client Header (Dễ bypass)

- **File:** [chat/route.ts](file:///home/haichu/tinnimate/web/app/api/chat/route.ts#L17)
- **Nguy hiểm:** Guest limit (2 tin nhắn) chỉ check header `X-Guest-Count` do client gửi lên. Hacker chỉ cần **không gửi header** hoặc **luôn gửi `0`** → dùng AI miễn phí không giới hạn, **tốn API Gemini tiền thật**.
- **Cách sửa:** Track guest usage bằng IP + fingerprint ở server-side, hoặc dùng cookie httpOnly, hoặc yêu cầu đăng nhập sau 2 tin.

### 3. SQL Injection Risk trong Admin User Search

- **File:** [admin/users/route.ts](file:///home/haichu/tinnimate/web/app/api/admin/users/route.ts#L29)
- **Nguy hiểm:** Dòng `query.or(\`name.ilike.%${search}%,email.ilike.%${search}%\`)` nối trực tiếp user input vào query string. Mặc dù Supabase PostgREST có một lớp bảo vệ, nhưng string interpolation kiểu này vẫn có thể bị khai thác với các ký tự đặc biệt.
- **Cách sửa:** Sanitize `search` input: loại bỏ `%`, `_`, và các ký tự đặc biệt trước khi đưa vào query. Hoặc dùng Supabase filter API an toàn hơn.

---

### 4. Hardcoded Admin Email Whitelist

- **File:** [require-admin.ts](file:///home/haichu/tinnimate/web/lib/supabase/require-admin.ts#L4-L10)
- **Nguy hiểm:** 5 email admin được hardcode trong source code. Nếu source code bị lộ (ví dụ push lên public repo), attacker biết chính xác email nào có quyền admin. Thêm nữa, không thể thay đổi admin list mà không deploy lại.
- **Cách sửa:** Chuyển admin emails vào database hoặc env variable. Giữ email whitelist chỉ làm fallback khi DB chưa setup.

---

### 5. `dangerouslySetInnerHTML` trong Chat Page (XSS Potential)

- **File:** [chat/page.tsx](file:///home/haichu/tinnimate/web/app/(app)/chat/page.tsx#L38)
- **Nguy hiểm:** Sử dụng `dangerouslySetInnerHTML` render nội dung mà không sanitize có thể dẫn đến XSS (hacker chèn code độc). Dù hiện tại data đến từ i18n (an toàn), nhưng nếu sau này data đổi nguồn → nguy hiểm.
- **Cách sửa:** Dùng DOMPurify hoặc thư viện sanitize HTML trước khi render. Hoặc refactor tránh `dangerouslySetInnerHTML`.

### 2. Pricing Config Route dùng Service Role Key cho Public Endpoint

- **File:** [pricing-config/route.ts](file:///home/haichu/tinnimate/web/app/api/pricing-config/route.ts#L6)
- **Hậu quả:** Public endpoint (không cần auth) nhưng lại dùng `createServiceClient()` — service role key bypass RLS hoàn toàn. Dù đã strip gateway credentials trước khi trả về (tốt), nhưng nếu code logic bị sửa sai → lộ secret keys.
- **Cách sửa:** Dùng `createClient()` (anon key) thay vì service client cho public routes. Cấu hình RLS cho phép public read `admin_config.pricing_config`.
